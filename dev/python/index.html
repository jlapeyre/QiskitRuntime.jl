Using PythonCall and other packages

There are a few ways to try to conveniently use packages that are not dependencies of QiskitRuntime. (Or of any other package you want to use)

You can dump all of the desired packages in your default environment. But this is widely discouraged, for good reasons. If you are relatively new to Julia, you will surely ignore this advice and add packages in whatever environment you happen to be in. Eventually, version conflicts, long load and compile times, and failed compilation will become more trouble than the convenience is worth.

I stopped writing this after the previous sentence and wrote QiskitRuntime.ExtraEnvs, which is meant to manage extra packages. The most important is PythonCall for communicating with Python. I'll explain how to use it below.

PythonCall for Python langauge Qiskit packages

We want to use the Python language packages qiskit (the Qiskit SDK) and qiskit-ibm-runtime (the Qiskit Runtime IBM Client).

Setting up a Python environment

PythonCall can completely manage your Python environment, installing Python and required packages using its own robust system, avoiding things like pip altogether. I am not using this at the moment; for a couple of reasons.

  • The last time I tried this, I found it rather heavy and obtrusive. For example, it silently and without prompting began to download a Python distribution, binary and all. Howver, my imperfect understanding is that in the meantime, the author has added more flexibility and ergonomics.
  • QiskitRuntime is not meant to completely hide the Python from the user and only expose Julia. In fact most people using QiskitRuntime will be very familiar with qiskit and qiskit-ibm-runtime. And they know how to manage a Python environment that includes these packages and various other supporting packages particular to the user.

Howver, I think it's a good idea in the future to provide the configuration files in order to do it the PythonCall way.

So for now, I assume you use something like

  • python -m venv ./venvs/py312 to set up an environment.
  • activate the environment
  • add at least qiskit and qiskit-ibm-runtime to the environment.
  • Start a Julia REPL (or notebook, etc) from a shell with the Python environment activated.

In order to prevent PythonCall from opaquely installing Python and a bunch of other stuff somewhere (you can look up where in the docs) set the environment variable JULIA_CONDAPKG_BACKEND. For example, from within Julia:

ENV["JULIA_CONDAPKG_BACKEND"] = "Null"

Now you can load PythonCall as follows (or use another method if you prefer)

Read the file ./extra/extras.jl. Add or remove packages from the list. Then do include("./extra/extras.jl"). The packages in the list will then be available without polluting any Project.tomls or dependency lists. Furthemore, PythonCall will be loaded, and the extension ./ext/PythonExt.jl will be loaded as well.

You should be able to do this (depending on if and how you edit extra.jl).

julia> using Revise; using QiskitRuntime;

julia> include("extra/extra.jl");

julia> circ = qk().QuantumCircuit(2); circ.cx(0, 1); print(circ)

q_0: ──■──
     ┌─┴─┐
q_1: ┤ X ├
     └───┘

Using qr() and qk() is not super convenient. But you can't do using QiskitRuntime.SomeModule: qk, qr to get handles to the modules. Still there could be an easier way.
